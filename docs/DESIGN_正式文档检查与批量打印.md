# 批量打印：正式文档检查 + 封面签字检查 方案设计

## 一、整体流程

```
┌─────────────────────────────────────────────────────────────────┐
│  批量打印流程                                                     │
├─────────────────────────────────────────────────────────────────┤
│  第一步：正式性检查（是否可视为正式文档）                          │
│    → 不通过：列出问题，禁止打印（或由用户选择“仍要打印”）           │
│  第二步：封面签字与日期检查                                        │
│    → 不通过：列出缺失项，禁止打印（或由用户选择“仍要打印”）         │
│  第三步：可选自动修复（接受修订、去除高亮、删除批注等）             │
│  第四步：提交打印                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 二、第一步：正式性检查（正式文档判定）

**原则**：以下任一存在，则视为**非正式文档**，需提示并可选拦截打印。

| 检查项 | 说明 | 实现方式（Word） | Excel/PDF |
|--------|------|------------------|-----------|
| **标黄/高亮** | 文字或单元格存在高亮（如黄色） | `Range.HighlightColorIndex != wdNoHighlight` 遍历 StoryRanges | `Interior.ColorIndex` 为黄/浅黄等 |
| **修订记录** | 存在未接受的“跟踪的更改” | `Revisions.Count > 0` | 共享工作簿的跟踪修订 |
| **批注/评论** | 文档中存在批注 | `Comments.Count > 0` | 单元格批注 |
| **非黑字体** | 正文或封面关键区域使用非黑色字体 | `Font.Color != wdColorBlack` 抽样或全量 | `Font.Color != 0` |
| **隐藏文字** | 存在“隐藏”格式的文字 | `Font.Hidden == True` | N/A |
| **草稿水印** | 页眉/页脚中存在“草稿”“DRAFT”等水印 | 检测 HeaderFooter 内 Shape 文本 | N/A |
| **未解析的域** | 存在未更新的域（如日期域未更新） | 可选：检测 `Fields` 中未更新项 | N/A |

**推荐优先级**  
- **必做**：标黄、修订、批注、非黑字体（与当前需求一致，易实现、易解释）。  
- **建议**：隐藏文字、草稿水印（提升“正式感”，实现成本中等）。  
- **可选**：未解析的域（按你们是否大量用域决定）。

**输出**：返回“是否通过” + “问题列表”，例如：  
`{ "passed": false, "issues": ["存在 3 处修订", "存在 5 条批注", "存在非黑色字体"] }`

## 三、第二步：封面签字与日期检查

**原则**：封面页必须包含**电子签字**和**日期**，否则视为未完成审批，不允打印（除非用户强制跳过）。

当前实现已支持：  
- 从封面（第一页）文本中解析：**作者、审核、批准、日期**  
- 判断规则：各字段非空（且非纯占位符，如仅下划线、空格）

**可扩展点**（需你提供信息）：  
- 封面是否固定为**第一页**？是否有“封面在第二页”等模板？  
- 除 作者/审核/批准/日期 外，是否还有**必填项**（如“编制”“会签”“签发日期”）？  
- “电子签字”是否仅指**文本填写**，还是必须检测 Word **数字签名/签名行**（`Signatures`）？  
- 日期格式是否有要求（如必须 yyyy-mm-dd 或中文日期）？是否要做**合法性校验**（如不能是未来日期）？

**输出**：返回“是否通过” + “缺失项列表”，例如：  
`{ "passed": false, "missing": ["批准", "日期"] }`

## 四、需要您提供的信息

请尽量按下面几条给出说明，便于把方案落到你现有模板和流程上。

### 1. 文档类型与范围

- 是否只做 **Word**，还是也要支持 **Excel / PDF**？  
- 若支持 Excel：正式性检查是否只做“修订 + 高亮 + 批注”，封面签字是否在 Excel 第一张表？

### 2. 封面与签字规范

- 封面**固定在第几页**？是否总是第一页？  
- 必填的**签字/日期项**有哪些？（例如：作者、审核、批准、日期；是否还有“编制”“会签”等）  
- 这些项在文档里是**固定关键词**（如“作者：”“审核：”）还是**表格/控件**？若是表格，能否提供一份**样例文档**或截图（可脱敏）？  
- “电子签字”的认定方式：  
  - 仅“有文字即可”？  
  - 还是必须检测 Word 的**签名行/数字签名**（Signatures）？

### 3. 正式性检查的严格程度

- 是否**必须**包含：标黄、修订、批注、非黑字体、隐藏文字、草稿水印？  
- 是否有**行业/公司规范**（如必须无批注、必须无修订）需要写进说明或配置？

### 4. 交互与策略

- 检查不通过时：  
  - **禁止打印**，仅提示问题？  
  - 还是允许用户勾选“**仍要打印**”（并记日志/留痕）？  
- 是否需要在**自动修复**后再做一次正式性检查（避免修复不完整）？

### 5. 环境与部署

- 是否只在 **Windows + 本机 Office** 上运行？是否有计划支持 **Linux/无 Office 环境**（若支持，需改用 python-docx 等，且无法做打印与部分 COM 检查）？  
- 打印是否**仅本机**，还是需要考虑**网络打印机/队列**？

## 五、推荐实现方案（与现有项目对接）

1. **新增“正式性检查”模块**  
   - 输入：文件路径（及类型）。  
   - 输出：`passed: bool`、`issues: list[str]`。  
   - 内部按顺序检查：高亮 → 修订 → 批注 → 非黑字体 →（可选）隐藏文字、水印。  
   - 与现有 `word_handler` / `excel_handler` 复用同一 COM 打开逻辑，尽量**一次打开、多项目检查**，减少重复打开文件。

2. **保持“封面签字检查”独立**  
   - 沿用现有 `check_cover_signature`，根据你反馈扩展关键词与必填项。  
   - 若需支持“签名行/数字签名”，再在 Word 里增加对 `Signatures` 的检查。

3. **流程串联**  
   - 先跑**正式性检查** → 再跑**封面签字检查** → 若都通过（或用户选择“仍要打印”），再执行现有逻辑：接受修订、去高亮、统一黑色、打印。  
   - 可选：在“自动修复”后对 Word 再跑一次正式性检查，确保无遗漏。

4. **配置与扩展**  
   - 将“必检项”“封面关键词”“是否允许强制打印”等做成**配置**（如 JSON 或环境变量），便于你后续增删检查项或调整策略。

## 六、小结

| 步骤 | 内容 | 需要你提供 |
|------|------|------------|
| 第一步 | 正式性检查：标黄、修订、批注、非黑字体等 | 是否加隐藏文字/水印/域；Excel、PDF 范围 |
| 第二步 | 封面签字与日期检查 | 封面页位置、必填项列表、是否校验签名行/日期格式 |
| 交互 | 不通过时禁止或允许“仍要打印” | 策略选择 |
| 部署 | 运行环境与打印方式 | Windows + Office 是否唯一；是否仅本机打印 |

按上述信息补充后，可以在现有 `batch_print` + `word_handler` / `excel_handler` 上直接落实现有两阶段检查逻辑，并预留配置与扩展点。

## 七、当前已实现情况

- **第一步 正式性检查**（Word）：`word_handler.check_formal_document()`  
  - 检查：修订数、批注数、标黄高亮、非黑字体、可选隐藏文字。  
  - 返回 `(passed, issues)`，已接入 `batch_print`（`check_formal` 参数）与 Web 勾选项。
- **第一步 正式性检查**（Excel）：`excel_handler.check_formal_document()`  
  - 检查：单元格标黄/高亮、单元格批注。  
  - 已接入批量流程。
- **第二步 封面签字检查**（Word）：`word_handler.check_cover_signature()`  
  - 检查第一页关键词：作者、审核、批准、日期，均已接入。
- 命令行：`--no-formal-check` 可关闭正式性检查；Web 页有「第一步：正式性检查」勾选项。
